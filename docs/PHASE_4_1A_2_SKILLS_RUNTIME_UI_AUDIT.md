## 2. What data already reaches the client per ability (before this phase)

From `mapQuickAction` (abilityRuntimeMapper.js), today's shape has: `characterActionId`, `definitionId`, `sourceType`, `type`, `name`, `shortDescription`, `fullDescription`, `iconKey`, `semanticKind`, `targeting{mode,minTargets,maxTargets,allowAllies,allowSelf,requiresBodyZone}`, `costs{main,move,psi,charges}`, `cooldown{current,max,unit,active}`, `state{available,active,disabledReason,selectable}`, `requirements{weaponClass,weaponId,conditionSummary}`.

**Gaps found**, all server-side (migration 92), none client-invented-around today:
- `state.available` factors in `is_enabled` / `skip_turn` / `alive-or-target-none` — but **not** `cooldown.current > 0` and **not** resource (PSI/charges) sufficiency. A technique on cooldown or lacking PSI today reports `available: true` from the RPC; only its `disabledReason` text happens to mention cooldown (never resource insufficiency — there is no resource check in the SQL at all yet).
- There is no `executionAvailable`/`executionReason` field anywhere. Nothing today tells the client "this specific attack technique's effect (damage/armor-pierce/ignore-armor) isn't implemented on the weapon-attack path" — a client that wanted to warn about that pre-attack would have had to hardcode a check on the ability's name or a client-side effect-grammar copy, exactly what §B of this phase forbids.
- `ald` (ability_level_defs) is **already joined** in the existing quickbar query for cost/cooldown display — it already carries `attack_damage_bonus`/`attack_armor_pierce`/`ignore_armor`/`resource_cost`, the exact same columns migration 100's `perform_attack` checks to reject `ACTION_EFFECT_NOT_IMPLEMENTED`. No new join is needed for the unsupported-effect check; only for the PSI-pool sufficiency check (`odyssey_character_resource_pools`/`odyssey_resource_pool_defs`, not currently joined in this function).
- **Known, deliberately-not-fixed pre-existing imprecision** (documented, not touched here): the quickbar query resolves `ald` at `ability_level = ca.learned_level` directly, while `odyssey_get_character_ability_effective_level` (migration 40's redefinition) can return a HIGHER effective level via a linked skill's progression. `perform_attack` (migration 100) uses the true effective level; this quickbar RPC has used the simpler `learned_level` join since Phase 4.0 for cost/cooldown display and continues to for the new fields too, for internal consistency with the rest of this same function — fixing that mismatch is a separate, broader task, not in scope here.

## 3. Reusable detail card confirmed — no second parallel UI built

`QuickbarEditorPanel.js`'s `renderDescriptionPanel()` is a complete ability detail card (icon + name + category header, description paragraph, cost/cooldown/target metadata pills, status banner) built entirely from `abilityTooltipModel(action)`. This phase extracts its action-rendering body into `hud/abilities/AbilityDetailCard.js` (a small pure function taking an `action` — no editor-specific `resolved`/selection wrapper), and has both the Quickbar Editor and the new Skills Block hover/click detail card call it. The Quickbar Editor's own placeholder states (`Select an ability...`, `Empty slot...`, `Missing action...`) stay in `QuickbarEditorPanel.js` — those are editor-specific and out of scope.
